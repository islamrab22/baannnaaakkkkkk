# Bank of Palestine ‚Äî Banking Platform & Admin CMS

A production-grade banking website with a full Admin CMS, built on a clean-architecture
Express/TypeScript backend and a React/Vite frontend.

## Stack

- **Frontend:** React 19, Vite, TypeScript, Tailwind CSS v4, React Router, Recharts, Sonner
- **Backend:** Express, TypeScript, clean architecture (controllers/services/repositories)
- **Database:** PostgreSQL via Prisma ORM
- **Auth:** JWT access + refresh tokens (httpOnly cookie rotation), bcrypt, role-based access (Admin/Editor/Viewer)
- **Uploads:** Cloudinary (URLs only persisted in Postgres)
- **Security:** Helmet, CORS, rate limiting, Zod validation, XSS sanitization, audit logging

## Project Structure

```

  config/           env, prisma client, cloudinary, logger, static content
  controllers/       Route handlers

  middleware/        auth, rbac, validation, error handling, upload, rate limit

  admin/              Admin CMS SPA, mounted at /admin
```

## Local Development

**Prerequisites:** Node.js 20+, PostgreSQL 14+

1. Install dependencies:
   ```
   npm install
   ```
2. Copy `.env.example` to `.env` and fill in `DATABASE_URL`, `DIRECT_URL` (same value as
   `DATABASE_URL` for a plain local/Docker Postgres ‚Äî see the Render section below for why
   they differ when using a connection pooler like Supabase), JWT secrets, and (optionally)
   Cloudinary credentials.
3. Run migrations and generate the Prisma client:
   ```
   npm run prisma:migrate
   ```
4. Seed the database with demo content and staff accounts:
   ```
   npm run prisma:seed
   ```
   This prints the generated admin/editor/viewer credentials to the console.
5. Start the dev server (Express + Vite middleware):
   ```
   npm run dev
   ```
6. Visit `http://localhost:3000` for the public site and `http://localhost:3000/admin`
   for the CMS.

## Roles

## Deployment

### Render (backend + API, via Docker)

Render is configured to build and run this service from the `Dockerfile` (see `render.yaml`,
`runtime: docker`). The image's `CMD` runs `npm run start:render`, i.e.
`scripts/start-render.sh`, which applies pending migrations (`prisma migrate deploy`) and then
starts the server bound to `process.env.PORT`.

In the Render dashboard, choose **New > Blueprint** and point it at this repo to create the
service with the correct Dockerfile path and health check (`/api/health`) already wired up.
You still need to fill in the values Render can't invent for you (`sync: false` entries):
`DATABASE_URL`, `DIRECT_URL`, `CLIENT_URL`, and the `CLOUDINARY_*` keys. `JWT_ACCESS_SECRET`,
`JWT_REFRESH_SECRET`, and `COOKIE_SECRET` are auto-generated by Render.

**Do not** set a `PORT` environment variable yourself ‚Äî Render injects it automatically at
runtime, and the server already binds to `process.env.PORT` (see `server/config/env.ts` /
`server.ts`).

#### Database connection pooling (read this if the deploy hangs / "No open ports detected")

There are two distinct failure modes that both surface as Render's generic
**"No open ports detected"**:

1. **Missing/invalid env vars.** If `DATABASE_URL`, `JWT_ACCESS_SECRET`, or `JWT_REFRESH_SECRET`
   aren't set, the process fails env validation and exits *before* it ever calls `app.listen` ‚Äî
   check the logs for an `Invalid environment configuration` message.
2. **`prisma migrate deploy` hangs against a pooled connection.** If `DATABASE_URL` or
   `DIRECT_URL` points at a *transaction-mode* connection pooler (PgBouncer, Supabase's
   Supavisor pooler on port 6543, etc.), `prisma migrate deploy` can hang indefinitely with
   **no error and no further log output** ‚Äî it needs a session-level Postgres advisory lock,
   and a transaction pooler can route the lock and unlock statements to different backend
   connections, so the unlock never arrives. From Render's side this looks identical to a slow
   boot until its own deploy timeout fires and it reports no open port. This is why the schema
   declares a separate `directUrl`:
   ```prisma
   datasource db {
     url       = env("DATABASE_URL")  // pooled connection is fine for the app itself
     directUrl = env("DIRECT_URL")    // must be a direct, non-pooled connection
   }
   ```
   On **Supabase**: set `DATABASE_URL` to the **Transaction pooler** connection string
   (port `6543`) and `DIRECT_URL` to the **Direct connection** string (port `5432`). Only
   `DIRECT_URL` is used by the Prisma CLI (`migrate deploy`/`migrate dev`); the app itself uses
   `DATABASE_URL` at runtime.

   As a safety net, `scripts/start-render.sh` wraps `prisma migrate deploy` in a
   `MIGRATE_TIMEOUT_SECONDS` timeout (default 60s). If it's ever hit, the container now fails
   fast with a clear `[start-render] FATAL: 'prisma migrate deploy' timed out...` log line
   pointing at this exact issue, instead of hanging silently for the platform's full deploy
   timeout.

If you split the frontend to **Vercel** instead of serving it from the same Render service,
point its API calls at the Render backend URL and set `CLIENT_URL` on the backend to the
Vercel origin (CORS).

- **Database:** **Supabase PostgreSQL** ‚Äî see the pooling note above for `DATABASE_URL` vs.
  `DIRECT_URL`.
- **Images:** **Cloudinary** ‚Äî set `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET`.

### Docker

A `Dockerfile` and `docker-compose.yml` are included for local or self-hosted deployment:

```
docker compose up --build
```

This starts a Postgres container and the app container (which runs `prisma migrate deploy`
on boot). Set `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`, and `COOKIE_SECRET` in your shell
or a `.env` file before running in anything beyond local testing.

## Security Notes

- Access tokens are short-lived JWTs sent in the `Authorization` header; refresh tokens are
  rotated on each use and stored hashed, in an httpOnly cookie scoped to `/api/auth`.
- All admin-facing routes require authentication; write operations additionally require the
  `EDITOR` or `ADMIN` role, and destructive operations require `ADMIN`.
- All mutating input is validated with Zod and sanitized against XSS before persistence.
- Every admin create/update/delete/login action is written to the `audit_logs` table.
